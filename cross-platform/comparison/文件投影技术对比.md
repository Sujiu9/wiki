# 跨平台文件投影技术对比：FileProvider vs ProjFS

## 为什么需要文件投影（Projected File System）？

文件投影的核心需求：**让进程看到一个虚拟的文件系统视图，实际内容由用户态程序按需提供**。

典型场景：
- AI Agent 沙盒：Agent 看到完整文件树，修改写入隔离层
- 云存储客户端：OneDrive/iCloud 按需下载，本地只显示占位符
- 大型代码仓库：VFS for Git，百万文件按需 hydrate
- 容器：overlay2 文件系统，镜像层只读 + 容器层读写

---

## 平台方案总览

| 平台 | 技术 | 提供者 | 引入版本 |
|------|------|--------|----------|
| macOS | FileProvider Extension | Apple | macOS 11 (2020) |
| Windows | ProjFS (Projected File System) | Microsoft | Windows 10 1803 (2018) |
| Linux | FUSE + OverlayFS | 社区 | 内核 3.18+ (2014) |

## 核心架构对比

### macOS FileProvider

```
┌─────────────────────────────────────────────────────┐
│  应用进程                                            │
│  open("/path/to/projected/file.txt")                 │
└───────────────────────┬─────────────────────────────┘
                        │ VFS 请求
┌───────────────────────▼─────────────────────────────┐
│  fileproviderd (系统守护进程)                          │
│  • 管理 Extension 生命周期                            │
│  • 缓存策略                                          │
│  • 枚举调度                                          │
└───────────────────────┬─────────────────────────────┘
                        │ 回调
┌───────────────────────▼─────────────────────────────┐
│  FileProvider Extension (独立用户进程)                 │
│  提供: 枚举 / 内容 / 元数据                           │
└─────────────────────────────────────────────────────┘
```

### Windows ProjFS

```
┌─────────────────────────────────────────────────────┐
│  应用进程                                            │
│  CreateFile("C:\\projected\\file.txt")               │
└───────────────────────┬─────────────────────────────┘
                        │ I/O 请求
┌───────────────────────▼─────────────────────────────┐
│  ProjFS 文件系统过滤驱动 (prjflt.sys)                 │
│  • 内核态运行                                        │
│  • 拦截 I/O 请求                                     │
│  • 管理占位符/水合状态                                │
└───────────────────────┬─────────────────────────────┘
                        │ 回调 (用户态)
┌───────────────────────▼─────────────────────────────┐
│  ProjFS Provider (用户态程序)                         │
│  实现: PRJ_START_DIRECTORY_ENUMERATION_CB            │
│        PRJ_GET_PLACEHOLDER_INFO_CB                   │
│        PRJ_GET_FILE_DATA_CB                          │
└─────────────────────────────────────────────────────┘
```

### Linux OverlayFS + FUSE

```
┌─────────────────────────────────────────────────────┐
│  应用进程                                            │
│  open("/merged/file.txt")                            │
└───────────────────────┬─────────────────────────────┘
                        │ VFS
┌───────────────────────▼─────────────────────────────┐
│  OverlayFS (内核文件系统)                             │
│  upperdir (读写层) + lowerdir (只读层)                │
│  或 FUSE daemon (用户态文件系统)                      │
└─────────────────────────────────────────────────────┘
```

## 详细特性对比

| 特性 | macOS FileProvider | Windows ProjFS | Linux OverlayFS |
|------|-------------------|----------------|-----------------|
| 按需加载 | ✅ 枚举后按需 fetch | ✅ 占位符 → hydrate | ❌ 全量挂载 |
| 写时复制 | ✅ (需自行实现 overlay) | ✅ (内核支持 dirty 标记) | ✅ (内核原生) |
| 删除追踪 | 需自行维护 deletions | ✅ 内核自动追踪 tombstone | ✅ whiteout 文件 |
| 目录枚举 | Extension 回调枚举 | Provider 回调枚举 | 联合视图自动合并 |
| 内容提供 | `fetchContents(for:)` | `PRJ_GET_FILE_DATA_CB` | 直接读 lower/upper |
| 进程托管 | 系统进程管理 (fileproviderd) | 自行管理 | 内核/自行管理 |
| 内存限制 | ~80MB（系统强制） | 无特殊限制 | N/A |
| 性能 | 中（进程间通信开销） | 高（内核驱动直接回调） | 高（纯内核操作） |
| 缓存 | 系统自动缓存（需 signal 刷新） | 无中间缓存 | 无 |
| 稳定性 | 高（系统保活） | 高 | 极高（纯内核） |

## 文件状态模型对比

### macOS FileProvider

```
                  ┌──────────────┐
                  │  dataless    │  ← 仅元数据，无内容
                  │  (占位符)     │
                  └──────┬───────┘
                         │ fetchContents
                  ┌──────▼───────┐
                  │ materialized │  ← 内容已下载到本地
                  │  (已物化)     │
                  └──────┬───────┘
                         │ 用户修改
                  ┌──────▼───────┐
                  │   dirty      │  ← 本地有未同步的修改
                  │  (已修改)     │
                  └──────────────┘
```

### Windows ProjFS

```
     ┌────────────┐      ┌──────────────┐      ┌────────────┐
     │  Virtual   │ ───► │ Placeholder  │ ───► │ Full/Dirty │
     │ (纯虚拟)   │      │ (有元数据)    │      │ (完整/修改) │
     └────────────┘      └──────────────┘      └────────────┘
     目录枚举触发           读取内容触发          写入触发
```

### Linux OverlayFS

```
读取: merged → 先查 upper → 无则查 lower
写入: copy-up（从 lower 复制到 upper）→ 写入 upper
删除: 在 upper 创建 whiteout 标记文件（字符设备 0,0）
```

## 实践中的关键差异

### 1. 谁负责 Overlay？

| 平台 | Overlay 实现 |
|------|-------------|
| macOS | **应用自行实现**——FileProvider 只提供枚举/内容回调，COW overlay 逻辑需自己写 |
| Windows | **内核辅助**——ProjFS 自动追踪文件的 dirty 状态，但 overlay 合并仍需应用处理 |
| Linux | **内核完整实现**——OverlayFS 原生提供 upper/lower 分层和 whiteout |

### 2. 删除如何表达？

```
macOS:    应用维护 deletions.json 集合
Windows:  ProjFS tombstone + 应用侧 deletions
Linux:    内核自动创建 whiteout 文件（mknod c 0 0）
```

### 3. 目录创建追踪

```
macOS:    需要自行维护 created_dirs/ 标记目录
Windows:  ProjFS 自动识别新目录（Placeholder vs Full）
Linux:    upper 目录中直接可见新目录
```

## 统一抽象层设计

跨平台沙盒需要在两套底层 API 上建立统一抽象：

```rust
/// 跨平台文件投影接口
trait ProjectionProvider {
    /// 启动投影，指定源目录和挂载点
    async fn start(&self, source: &Path, mount_point: &Path) -> Result<()>;
    
    /// 停止投影
    async fn stop(&self) -> Result<()>;
    
    /// 获取变更列表
    fn list_changes(&self) -> Vec<FileChange>;
    
    /// 将变更写回源目录
    async fn apply_changes(&self) -> Result<()>;
    
    /// 丢弃所有变更，恢复为源目录状态
    async fn discard_changes(&self) -> Result<()>;
}

enum FileChange {
    Created(PathBuf),
    Modified(PathBuf),
    Deleted(PathBuf),
}
```

平台实现：
- macOS: `FileProviderProjection` → NSFileProviderManager + WholeDiskOverlay
- Windows: `ProjFSProjection` → ProjFS API + overlay 追踪

## 性能考量

| 操作 | macOS FileProvider | Windows ProjFS |
|------|-------------------|----------------|
| 首次枚举 10K 文件 | ~2s（Extension 进程间通信） | ~0.5s（内核直接回调） |
| 读取已缓存文件 | ~0.1ms（fileproviderd 缓存） | ~0.05ms（内核直读） |
| 读取未缓存文件 | ~5ms（回调 Extension） | ~2ms（回调 Provider） |
| 写入文件 | ~1ms（直写 overlay） | ~0.5ms（直写磁盘） |

macOS 的主要瓶颈在于 fileproviderd 的中间层和 Extension 的进程间通信开销。
