# 🟡 跨平台进程生命周期管理

> 📋 前置知识：[macOS 进程与线程模型](../../macos/01-fundamentals/进程与线程模型.md)、[IPC 通信机制](./IPC通信机制.md)

## 为什么需要关注进程生命周期？

AI Agent 启动的命令可能运行数秒到数小时。正确管理子进程生命周期要解决三个核心问题：

1. **如何终止**：用户取消任务时，如何杀掉整棵进程树（不留孤儿）
2. **如何等待**：如何知道子进程已退出并获取退出码（不产生僵尸进程）
3. **如何协调 I/O**：确保先收完所有输出再报告退出（避免数据丢失）

## 进程终止对比

### macOS/Unix：信号机制

```
信号投递路径：

  kill(pid, SIGTERM)
       │
       ▼
  ┌─────────────────────────────────────────────────────┐
  │ 内核信号投递                                          │
  │                                                     │
  │  1. 找到目标 proc_t                                  │
  │  2. 设置 pending signal bitmap                      │
  │  3. 如果线程在可中断睡眠 → 唤醒它                     │
  │  4. 如果线程在用户态运行 → 下次返回用户态时处理        │
  └─────────────────────────────────────────────────────┘
       │
       ▼ (目标进程返回用户态时)
  ┌─────────────────────────────────────────────────────┐
  │ 信号处理                                             │
  │                                                     │
  │  检查信号处理器：                                     │
  │  • SIG_DFL → 执行默认动作（SIGTERM=终止）            │
  │  • SIG_IGN → 忽略                                  │
  │  • handler → 调用用户注册的处理函数                   │
  │                                                     │
  │  特殊：SIGKILL/SIGSTOP 不可被捕获或忽略！            │
  └─────────────────────────────────────────────────────┘
```

### Windows：TerminateProcess

```
终止路径：

  TerminateProcess(hProcess, exit_code)
       │
       ▼
  ┌─────────────────────────────────────────────────────┐
  │ 内核终止流程 (NtTerminateProcess)                     │
  │                                                     │
  │  1. 终止进程内所有线程                                │
  │  2. 关闭所有句柄                                     │
  │  3. 释放地址空间                                     │
  │  4. 设置退出码                                       │
  │  5. 进程对象变为 signaled 状态                        │
  └─────────────────────────────────────────────────────┘
```

### 对比

| 维度 | macOS (信号) | Windows (TerminateProcess) |
|------|-------------|---------------------------|
| 优雅退出 | SIGTERM（可捕获清理） | 无标准方案（需自定义 IPC） |
| 强制终止 | SIGKILL（不可捕获） | TerminateProcess（不可拦截） |
| 进程组批量终止 | `kill(-pgid, sig)` | 无原生支持，需遍历进程树 |
| 子进程继承 | 不继承信号处理器（exec 后重置） | N/A |
| 退出通知 | SIGCHLD 发给父进程 | 进程句柄变为 signaled |

## 进程树终止

### macOS：进程组语义

```rust
// AgentOS 的进程树终止实现
fn terminate_process_tree(pid: i32) {
    // 方案：利用 setsid() 创建的独立会话
    // 子进程启动时已经 setsid()，它是进程组 leader
    // 负 PID = 发送给整个进程组
    
    // 1. 先尝试优雅退出
    unsafe { libc::kill(-pid, libc::SIGTERM); }
    
    // 2. 等待一段时间
    std::thread::sleep(Duration::from_millis(100));
    
    // 3. 检查是否已退出
    let result = unsafe { libc::waitpid(pid, &mut status, libc::WNOHANG) };
    if result > 0 {
        return;  // 已退出
    }
    
    // 4. 强制终止
    unsafe { libc::kill(-pid, libc::SIGKILL); }
    
    // 5. 回收（防止僵尸进程）
    unsafe { libc::waitpid(pid, &mut status, 0); }
}
```

### Windows：遍历进程树

```rust
// Windows 没有进程组概念，需要手动遍历
fn terminate_process_tree_windows(root_pid: u32) {
    // 方案 1：taskkill /T /F（简单但依赖外部命令）
    Command::new("taskkill")
        .args(&["/T", "/F", "/PID", &root_pid.to_string()])
        .output();
    
    // 方案 2：NtQuerySystemInformation 遍历（纯 API）
    // 获取所有进程信息，通过 PPID 链找到所有子孙进程
    let snapshot = CreateToolhelp32Snapshot(TH32CS_SNAPPROCESS, 0);
    let mut children = Vec::new();
    // 递归收集所有 PPID == root_pid 的进程
    collect_descendants(snapshot, root_pid, &mut children);
    // 从叶子到根逐个终止
    for child in children.iter().rev() {
        TerminateProcess(child.handle, 1);
    }
}
```

## 僵尸进程与资源回收

### Unix 僵尸进程

```
进程退出后的状态：

  正在运行 ──── exit() ────► 僵尸态 (Zombie) ──── 父进程 waitpid() ────► 完全释放
                                    │
                                    │ 如果父进程不 waitpid()：
                                    │   进程表项永远不释放！
                                    │   （PID 被占用，/proc/<pid> 仍存在）
                                    │
                                    │ 如果父进程先退出：
                                    │   僵尸被 init (PID 1) 收养
                                    │   init 会自动 waitpid() 清理
                                    ▼
```

### 为什么僵尸进程有害？

```bash
# 每个僵尸进程占用：
# - 一个进程表项（struct proc）~ 几 KB
# - 一个 PID（系统 PID 数量有限，默认 max ~100000）
# 大量僵尸 → PID 耗尽 → fork() 返回 EAGAIN

$ ps aux | grep Z
USER  PID  %CPU %MEM  STAT COMMAND
root  1234  0.0  0.0  Z    [defunct]  ← 僵尸进程
```

### 正确的等待模式

```rust
// AgentOS 的等待实现

// 模式 1：阻塞等待（用于正常退出）
fn wait_blocking(pid: i32) -> i32 {
    let mut status: c_int = 0;
    unsafe { libc::waitpid(pid, &mut status, 0); }  // 阻塞直到退出
    
    if libc::WIFEXITED(status) {
        libc::WEXITSTATUS(status)     // 正常退出码 0-255
    } else if libc::WIFSIGNALED(status) {
        128 + libc::WTERMSIG(status)  // 信号终止：128 + 信号号
    } else {
        -1
    }
}

// 模式 2：非阻塞轮询（用于超时检查）
fn wait_nonblocking(pid: i32) -> Option<i32> {
    let mut status: c_int = 0;
    let result = unsafe { libc::waitpid(pid, &mut status, libc::WNOHANG) };
    match result {
        0 => None,           // 还在运行
        _ => Some(decode_status(status)),  // 已退出
    }
}

// Windows 等效
fn wait_windows(handle: HANDLE, timeout_ms: u32) -> Option<u32> {
    let wait_result = unsafe {
        WaitForSingleObject(handle, timeout_ms)
    };
    match wait_result {
        WAIT_OBJECT_0 => {
            let mut exit_code: u32 = 0;
            unsafe { GetExitCodeProcess(handle, &mut exit_code); }
            Some(exit_code)
        }
        WAIT_TIMEOUT => None,
        _ => None,
    }
}
```

## I/O 与退出的时序问题

### 核心问题

```
时间线：

  t0: 子进程 write("result\n") 到 stdout pipe
  t1: 子进程 exit(0)
  t2: 父进程收到 waitpid() 返回（进程已退出）
  t3: 父进程从 pipe 读到 "result\n"

  问题：如果先处理 t2 再处理 t3，会报告"进程退出"但丢失最后的输出！
```

### 解决方案：PipesEofBarrier

AgentOS 使用同步原语保证输出先于退出事件到达消费者：

```rust
/// 保证 processOutput 事件在 processExit 之前发送
struct PipesEofBarrier {
    remaining: Mutex<u32>,     // 还有多少个 pipe 未 EOF
    all_done: Condvar,         // 全部 EOF 时通知
}

impl PipesEofBarrier {
    fn new(pipe_count: u32) -> Self {
        Self {
            remaining: Mutex::new(pipe_count),  // 通常 = 2（stdout + stderr）
            all_done: Condvar::new(),
        }
    }
    
    /// pipe reader 线程在读到 EOF 时调用
    fn mark_eof(&self) {
        let mut remaining = self.remaining.lock().unwrap();
        *remaining -= 1;
        if *remaining == 0 {
            self.all_done.notify_all();
        }
    }
    
    /// 退出报告线程等待所有 pipe EOF
    fn wait_all_eof(&self, timeout: Duration) -> bool {
        let mut remaining = self.remaining.lock().unwrap();
        while *remaining > 0 {
            let result = self.all_done.wait_timeout(remaining, timeout).unwrap();
            remaining = result.0;
            if result.1.timed_out() {
                return false;  // 超时，强制继续
            }
        }
        true
    }
}

// 使用模式：
// 1. reader 线程持续读取 stdout/stderr，发送 processOutput 事件
// 2. reader 线程读到 EOF → barrier.mark_eof()
// 3. waiter 线程 waitpid() 得到退出 → barrier.wait_all_eof(5s)
// 4. 所有 pipe EOF 后 → 发送 processExit 事件
// 保证顺序：processOutput → ... → processOutput → processExit
```

## PTY vs ConPTY

### Unix PTY 架构

```
Agent 进程                          子进程 (bash)
┌──────────┐                       ┌──────────┐
│ master fd │◄═══ 内核 TTY 驱动 ═══►│ slave fd │
│  (读/写)  │     (line discipline) │ (stdin/  │
│           │                       │  stdout/ │
│           │                       │  stderr) │
└──────────┘                       └──────────┘

特性：
• isatty() = true → 程序表现为交互模式
• 行缓冲（不是全缓冲）
• Ctrl+C → 内核自动转为 SIGINT
• TIOCSWINSZ → 可动态调整窗口大小
```

### Windows ConPTY 架构

```
Agent 进程                                  子进程 (cmd.exe)
┌──────────┐                               ┌──────────┐
│ Input     │──► ┌──────────────────┐ ──►│ stdin    │
│ Pipe      │    │  ConPTY (conhost)│     │          │
│           │    │                  │     │          │
│ Output    │◄── │  • VT 序列解析    │ ◄──│ stdout   │
│ Pipe      │    │  • 窗口模拟      │     │          │
└──────────┘    └──────────────────┘     └──────────┘

API 调用链：
1. CreatePipe() × 2（输入管道 + 输出管道）
2. CreatePseudoConsole(size, hInput, hOutput, 0, &hPC)
3. 配置 STARTUPINFOEX 的 ProcThreadAttributeList
4. CreateProcess(..., EXTENDED_STARTUPINFO_PRESENT, ...)
```

### 对比

| 维度 | Unix PTY | Windows ConPTY |
|------|----------|----------------|
| 最低系统版本 | 所有 Unix | Windows 10 1809+ |
| API 复杂度 | `openpty()` + `fork()` | 5+ API 调用 |
| 信号/Ctrl+C | 内核 TTY 驱动自动处理 | 通过 VT 序列传递 |
| 窗口大小调整 | `ioctl(TIOCSWINSZ)` | `ResizePseudoConsole()` |
| 编码 | 取决于 locale（通常 UTF-8） | UTF-16 / Code Page |
| 性能 | 直接内核路径 | 经过 conhost.exe 中转 |
| 进程隔离 | slave fd 在子进程内 | conhost.exe 独立进程 |

## 退出码约定

### Unix 退出码

| 范围 | 含义 | 示例 |
|------|------|------|
| 0 | 成功 | `exit(0)` |
| 1-125 | 程序自定义错误 | `exit(1)` |
| 126 | 命令不可执行（权限） | `bash: ./script: Permission denied` |
| 127 | 命令未找到 | `bash: notexist: command not found` |
| 128+N | 被信号 N 终止 | 128+9=137 → SIGKILL |

### Windows 退出码

| 范围 | 含义 | 示例 |
|------|------|------|
| 0 | 成功 | `ExitProcess(0)` |
| 1+ | 程序自定义错误 | `ExitProcess(1)` |
| 0xC0000005 | ACCESS_VIOLATION | 空指针/越界 |
| 0xC000013A | CONTROL_C_EXIT | Ctrl+C |
| 1 (taskkill) | 被 TerminateProcess 强杀 | `taskkill /F` |

### AgentOS 统一退出码处理

```rust
// 跨平台退出码规范化
fn normalize_exit_code(raw: i32, platform: Platform) -> ExitInfo {
    match platform {
        Platform::MacOS | Platform::Linux => {
            if raw >= 128 {
                ExitInfo {
                    code: raw,
                    reason: format!("killed by signal {}", raw - 128),
                    signal: Some(raw - 128),
                }
            } else {
                ExitInfo { code: raw, reason: "normal".into(), signal: None }
            }
        }
        Platform::Windows => {
            if raw as u32 >= 0xC0000000 {
                ExitInfo {
                    code: raw,
                    reason: format!("exception 0x{:08X}", raw as u32),
                    signal: None,
                }
            } else {
                ExitInfo { code: raw, reason: "normal".into(), signal: None }
            }
        }
    }
}
```
