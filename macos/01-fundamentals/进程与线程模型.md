# 🟡 macOS 进程与线程模型

> 📋 前置知识：[macOS 基础概念](./README.md)

## 为什么需要理解进程模型？

在 macOS 上实现进程沙盒、子进程管理、信号投递时，必须理解 XNU 内核的**双层进程抽象**。macOS 不是纯 POSIX 系统——它在 BSD 进程模型之下还有一层 Mach 任务模型，两者职责不同：

- **Mach 层**：资源容器（内存、端口）
- **BSD 层**：POSIX 语义（PID、信号、文件描述符）

不理解这种分层，就会遇到"发了 SIGKILL 但子进程没死"、"fork 后子进程为什么还能访问沙盒外的文件"等困惑。

## XNU 双层进程架构

```
┌─────────────────────────────────────────────────────────┐
│                    用户态 (User Space)                    │
│                                                         │
│   ┌──────────┐  ┌──────────┐  ┌──────────┐            │
│   │ Thread A │  │ Thread B │  │ Thread C │  (Pthread)  │
│   └────┬─────┘  └────┬─────┘  └────┬─────┘            │
│        │              │              │                   │
├────────┼──────────────┼──────────────┼───────────────────┤
│        ▼              ▼              ▼                   │
│   ┌─────────────────────────────────────────────┐       │
│   │           BSD Process (proc_t)               │       │
│   │                                             │       │
│   │  • PID, PPID, PGID, SID                    │       │
│   │  • 文件描述符表 (fd table)                    │       │
│   │  • 信号处理器 (signal handlers)              │       │
│   │  • 凭证 (uid, gid, groups)                  │       │
│   │  • Sandbox profile 引用                     │       │
│   └──────────────────┬──────────────────────────┘       │
│                      │ 1:1 关联                          │
│   ┌──────────────────▼──────────────────────────┐       │
│   │           Mach Task (task_t)                 │       │
│   │                                             │       │
│   │  • 虚拟地址空间 (vm_map)                     │       │
│   │  • 端口命名空间 (port namespace)             │       │
│   │  • IPC 权利集合                             │       │
│   │  • 线程容器                                 │       │
│   │  ┌────────┐  ┌────────┐  ┌────────┐       │       │
│   │  │Thread A│  │Thread B│  │Thread C│ (Mach) │       │
│   │  └────────┘  └────────┘  └────────┘       │       │
│   └─────────────────────────────────────────────┘       │
│                    内核态 (Kernel Space)                   │
└─────────────────────────────────────────────────────────┘
```

### 关键对应关系

| BSD 概念 | Mach 概念 | 说明 |
|----------|-----------|------|
| Process (proc_t) | Task (task_t) | 1:1 绑定，BSD process 包裹 Mach task |
| POSIX Thread (pthread) | Mach Thread (thread_t) | 1:1 绑定 |
| PID | Task port | PID 通过 `task_for_pid()` 可获取 task port |
| Signal | Mach exception | 信号底层通过 Mach exception 机制实现 |
| File descriptor | Mach port (部分) | 某些 fd 底层是 Mach port 的封装 |

## fork() 与 exec() 的内核实现

### fork() 在 XNU 中做了什么

```c
// XNU 内核中 fork 的简化流程 (bsd/kern/kern_fork.c)
int fork1(proc_t parent, ...) {
    // 1. 创建新的 Mach task（复制虚拟地址空间 — COW 语义）
    task_t child_task = task_create_internal(parent->task, ...);
    
    // 2. 创建新的 BSD process 结构
    proc_t child = proc_alloc();
    child->p_pid = nextpid();
    child->p_ppid = parent->p_pid;
    
    // 3. 关键：复制 sandbox profile（不可逆的继承！）
    child->p_sandbox = sandbox_copy(parent->p_sandbox);
    
    // 4. 复制文件描述符表
    child->p_fd = fdcopy(parent->p_fd);
    
    // 5. 复制信号处理器
    child->p_sigacts = sigacts_copy(parent->p_sigacts);
    
    // 6. 在子 task 中创建第一个 Mach thread
    thread_create(child_task, &child_thread);
    
    return child->p_pid;
}
```

**关键点**：
- 虚拟地址空间是 **Copy-on-Write** 复制的（fork 瞬间不真正拷贝内存页）
- Sandbox profile **直接复制且不可移除**——这就是为什么 App Sandbox 会被子进程继承
- 文件描述符表是深拷贝的（子进程有独立的 fd 表，但底层 vnode 共享引用计数）

### exec() 在 XNU 中做了什么

```c
// XNU 内核中 execve 的简化流程 (bsd/kern/kern_exec.c)
int execve(proc_t p, const char *path, ...) {
    // 1. 打开可执行文件，验证 Mach-O 头
    vnode_t vp = namei(path);
    parse_machheader(vp, &imgp);
    
    // 2. 检查 sandbox profile 是否允许 process-exec
    if (sandbox_check(p, "process-exec", path) == DENY)
        return EACCES;  // ← App Sandbox 在这里拦截！
    
    // 3. 读取新二进制的 Entitlements（如果有）
    entitlements = read_code_signature_entitlements(vp);
    
    // 4. 销毁旧地址空间，加载新二进制
    vm_map_deallocate(p->task->map);
    load_machfile(vp, &imgp);  // dyld 负责链接
    
    // 5. 关键：sandbox profile 不会被 exec 清除！
    // p->p_sandbox 保持不变
    
    // 6. 但如果新二进制有自己的 Entitlements，会叠加新 profile
    if (entitlements->app_sandbox) {
        sandbox_apply_entitlements(p, entitlements);
        // 叠加！不是替换！
    }
    
    // 7. 重置信号处理器为默认
    sigacts_reset(p->p_sigacts);
    
    // 8. 关闭 close-on-exec 的 fd
    fdcloseexec(p->p_fd);
    
    return 0;
}
```

**关键结论**：
- exec **不能**摆脱已有的 sandbox profile
- exec **可能**叠加新的 profile（如果新二进制有 App Sandbox entitlement）
- 这就是为什么被 sandbox-exec 启动的进程，无论内部再 exec 什么，都受原始 profile 约束

## 进程组与会话（Session）

### 为什么需要进程组？

AI Agent 启动的子进程往往会再 fork 出孙进程（如 `bash -c "make && ./run"`），形成进程树：

```
Agent 主进程 (PID 100)
  └── sandbox-exec (PID 101)
        └── bash (PID 102)
              ├── make (PID 103)
              │     ├── gcc (PID 104)
              │     └── gcc (PID 105)
              └── ./run (PID 106)
```

如果只 kill PID 101，PID 102-106 会变成**孤儿进程**继续运行。必须使用**进程组**一次性终止整棵树。

### Unix 进程层次结构

```
Session (SID)
├── 前台进程组 (Foreground Process Group)
│   └── shell 及其管道命令
├── 后台进程组 1 (Background Process Group)
│   └── 后台任务 &
└── 后台进程组 2
    └── ...
```

| 概念 | ID | 系统调用 | 作用 |
|------|-----|---------|------|
| 进程 | PID | `getpid()` | 唯一标识进程 |
| 进程组 | PGID | `setpgid()` / `getpgid()` | 信号批量投递单位 |
| 会话 | SID | `setsid()` / `getsid()` | 控制终端绑定单位 |

### setsid() 的内核行为

```c
// 调用 setsid() 后：
pid_t setsid(void) {
    // 1. 创建新的 session（当前进程成为 session leader）
    current->session = new_session();
    
    // 2. 创建新的进程组（当前进程成为 group leader）
    current->pgrp = current->pid;
    
    // 3. 脱离控制终端
    current->ctty = NULL;
    
    // 返回新的 SID（等于当前 PID）
    return current->pid;
}
```

### 实战：AgentOS 的进程组管理

```rust
// AgentOS 项目中 PTY 子进程的启动（简化）
unsafe {
    let pid = libc::fork();
    match pid {
        0 => {
            // === 子进程 ===
            
            // 1. 创建新会话（脱离父进程的 session）
            libc::setsid();
            
            // 2. 设置 PTY slave 为控制终端
            libc::ioctl(slave_fd, libc::TIOCSCTTY, 0);
            
            // 3. 重定向 stdin/stdout/stderr 到 PTY
            libc::dup2(slave_fd, 0);
            libc::dup2(slave_fd, 1);
            libc::dup2(slave_fd, 2);
            
            // 4. exec sandbox-exec
            libc::execvp(program, args);
        }
        child_pid => {
            // === 父进程 ===
            // 记录子进程 PID，后续用于进程组终止
        }
    }
}

// 终止时：以进程组语义发送信号
fn kill_process_tree(pid: i32) {
    // 负数 PID = 发送到整个进程组
    unsafe { libc::kill(-pid, libc::SIGKILL); }
    // 等待回收，防止僵尸进程
    unsafe { libc::waitpid(pid, &mut status, 0); }
}
```

**为什么用 `kill(-pid, SIGKILL)` 而不是 `kill(pid, SIGKILL)`？**

| 方式 | 行为 | 效果 |
|------|------|------|
| `kill(pid, sig)` | 只发给单个进程 | 孙进程变为孤儿进程，继续运行 |
| `kill(-pgid, sig)` | 发给整个进程组 | 组内所有进程（包括孙进程）都收到信号 |
| `killpg(pgid, sig)` | 同上 | 更语义化的封装 |

## PTY 伪终端机制

### 为什么 Agent 需要 PTY？

普通 pipe 和 PTY 的行为差异巨大：

| 维度 | pipe (stdin/stdout) | PTY (伪终端) |
|------|--------------------|----|
| `isatty()` 返回值 | `false` | `true` |
| 程序行为 | 很多程序禁用颜色、进度条、交互 | 正常交互式行为 |
| 行缓冲 vs 全缓冲 | stdout 变为全缓冲（4KB） | 行缓冲（`\n` 即 flush） |
| Ctrl+C 传递 | 需要手动发 SIGINT | 终端驱动自动转换为 SIGINT |
| 窗口大小 | 不支持 | 支持 `TIOCSWINSZ` |

### PTY 架构

```
┌────────────────────────────────────────────────────────┐
│ 用户态                                                  │
│                                                        │
│  Agent 主进程              子进程 (bash/python)          │
│  ┌──────────┐             ┌──────────┐                │
│  │ 读写      │             │ stdin    │                │
│  │ master fd │◄═══════════►│ stdout   │ (slave fd)     │
│  │           │             │ stderr   │                │
│  └────┬─────┘             └──────────┘                │
│       │                                                │
├───────┼────────────────────────────────────────────────┤
│       ▼                                                │
│  ┌─────────────────────────────────────────────┐      │
│  │         TTY Line Discipline (termios)        │      │
│  │                                             │      │
│  │  • 行编辑（退格、删除、Ctrl-U）               │      │
│  │  • 信号生成（Ctrl-C → SIGINT）               │      │
│  │  • 回显控制                                  │      │
│  │  • 流控制（Ctrl-S / Ctrl-Q）                 │      │
│  └─────────────────────────────────────────────┘      │
│              内核态                                     │
└────────────────────────────────────────────────────────┘
```

### openpty 的内核实现

```c
// 简化的 openpty 流程
int openpty(int *amaster, int *aslave, ...) {
    // 1. 打开 PTY master（/dev/ptmx）
    *amaster = open("/dev/ptmx", O_RDWR);
    
    // 2. 解锁 slave 端
    grantpt(*amaster);   // 设置 slave 的权限
    unlockpt(*amaster);  // 解锁 slave 允许打开
    
    // 3. 获取 slave 设备路径（如 /dev/ttys001）
    char *slave_path = ptsname(*amaster);
    
    // 4. 打开 slave 端
    *aslave = open(slave_path, O_RDWR);
    
    return 0;
}
```

### async-signal-safe 问题

```rust
// 关键：所有字符串分配必须在 fork() 之前完成！
let program_cstr = CString::new(program).unwrap();  // ← fork 前分配
let args_cstrs: Vec<CString> = args.iter()
    .map(|a| CString::new(a.as_str()).unwrap())
    .collect();  // ← fork 前分配

unsafe {
    let pid = libc::fork();
    if pid == 0 {
        // 子进程中只允许调用 async-signal-safe 函数！
        // malloc/free 不安全（可能在 fork 时锁被持有）
        // 所以这里只能用预分配的 CString
        libc::execvp(program_cstr.as_ptr(), args_ptrs.as_ptr());
        libc::_exit(127);  // exec 失败用 _exit 而非 exit
    }
}
```

**为什么 fork 后不能 malloc？**

fork 只复制调用线程，但地址空间中可能有其他线程正持有 malloc 的内部锁。子进程中这些锁永远不会被释放 → 如果子进程调用 malloc 就会**死锁**。
