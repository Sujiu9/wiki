# 🟡 macOS 进程沙盒机制

> 📋 前置知识：[macOS 基础概念](../01-fundamentals/README.md)

## 为什么需要进程沙盒？

现代操作系统中，任何进程默认拥有当前用户的全部权限——可以读取所有文件、连接任何网络、执行任意程序。这在 AI Agent 场景下尤其危险：

- Agent 子进程可能被诱导读取 `~/.ssh/id_rsa` 私钥
- 恶意脚本可能修改 `~/.zshrc` 注入后门
- 未经审计的网络请求可能泄露敏感数据

**核心问题**：如何让子进程在受限环境中执行，只拥有完成任务所需的最小权限？

---

## 解决方案概览

| 方案 | 简述 | 适用场景 |
|------|------|----------|
| App Sandbox (Entitlements) | 通过 Entitlements 声明 App 权限 | Mac App Store 分发的应用 |
| sandbox-exec + SBPL | 通过 Profile 语言定义进程的系统调用权限 | 命令行进程/子进程隔离 |
| System Extension | 用户态驱动/网络扩展的沙盒隔离 | 驱动替代方案 |
| 容器化 (Docker) | 基于 Linux namespace 的完整隔离 | 服务器端，macOS 需 VM |

---

## 方案对比

| 维度 | App Sandbox | sandbox-exec + SBPL | Docker |
|------|-------------|---------------------|--------|
| 粒度 | 粗（文件/网络/硬件大类） | 细（单个系统调用级别） | 完整隔离 |
| 适用对象 | GUI 应用 | 任意进程 | 服务/批处理 |
| 配置方式 | Xcode Entitlements 声明 | 文本 Profile 文件 | Dockerfile |
| 动态性 | 编译时固定 | 运行时动态生成 | 运行时 |
| macOS 支持 | ✅ 官方推荐 | ✅ 但官方已标注 deprecated | 需 VM |
| 性能开销 | 极低 | 极低 | 较高（VM 层） |

**实践选择**：对于 AI Agent 子进程沙盒，`sandbox-exec + SBPL` 是最佳方案——粒度精细、运行时可定制、对子进程透明。

## sandbox-exec 具体实现

### 基本用法

```bash
# 使用内置 profile
sandbox-exec -n no-internet /bin/bash

# 使用自定义 profile 文件
sandbox-exec -f my-profile.sb /usr/bin/python3 script.py

# 使用内联 profile（通过 -p 参数）
sandbox-exec -p '(version 1)(deny default)(allow process-fork)' /bin/ls
```

### SBPL 语言（Seatbelt Profile Language）

SBPL 是一种 S-expression 语法的策略描述语言：

```scheme
(version 1)                           ; 必须，声明版本
(deny default)                        ; 默认拒绝所有操作
(allow process-fork)                  ; 允许 fork 子进程
(allow process-exec)                  ; 允许执行程序
(allow file-read*                     ; 允许读取文件
  (subpath "/usr")                    ; 限定 /usr 下
  (subpath "/Library"))               ; 限定 /Library 下
(allow file-write*
  (subpath "/tmp"))                   ; 只允许写 /tmp
(allow network-outbound)              ; 允许出站网络
```

### 核心操作类型

| 操作 | 含义 |
|------|------|
| `file-read*` | 读取文件（含 metadata） |
| `file-write*` | 写入文件 |
| `file-write-unlink` | 删除文件 |
| `process-exec` | 执行程序 |
| `process-fork` | 创建子进程 |
| `network-outbound` | 出站网络连接 |
| `network-inbound` | 入站网络连接 |
| `mach-lookup` | Mach 端口查找（XPC 通信） |
| `sysctl-read` | 读取 sysctl 变量 |
| `signal` | 发送信号 |

### 路径过滤器

```scheme
(subpath "/Users/dev/project")        ; 匹配路径及所有子路径
(literal "/etc/hosts")                ; 精确匹配
(regex #"^/tmp/work-[0-9]+/.*$")      ; 正则匹配
```

## 实战：AI Agent 沙盒 Profile 设计

以下是基于 AgentOS 项目提炼的最佳实践：

### 设计原则

```
1. deny default — 默认拒绝一切
2. 读权限用黑名单 — 默认允许读，deny 敏感路径
3. 写权限用白名单 — 只允许写工作目录和临时目录
4. 网络拦截交给更高层 — sandbox-exec 内全部放行，由 NetworkExtension 细控
5. 注入 session tag — 便于日志追踪
```

### 完整示例

```scheme
(version 1)
(debug deny)
;; 默认拒绝，注入 session tag 便于日志过滤
(deny default (with message "SANDBOX_TAG_abc123"))

;; 基础进程权限
(allow process-fork)
(allow signal (target same-sandbox))

;; 读权限：默认全开，deny 敏感路径
(allow file-read*)
(deny file-read*
  (subpath "/Users/dev/.ssh")          ; SSH 密钥
  (subpath "/Users/dev/.aws")          ; AWS 凭证
  (subpath "/Users/dev/.gnupg")        ; GPG 密钥
  (regex #"(?i).*/\.env$")            ; .env 文件（大小写不敏感）
  (regex #"(?i).*/\.gitconfig$"))     ; git 配置

;; 写权限：仅工作目录 + 临时目录
(allow file-write*
  (subpath "/Users/dev/project")       ; 工作目录
  (subpath "/tmp")
  (subpath "/private/tmp")
  (subpath "/dev"))

;; 执行权限：全开，deny 危险工具
(allow process-exec)
(deny process-exec
  (literal "/usr/bin/sandbox-exec")    ; 防止嵌套沙盒绕过
  (literal "/usr/sbin/dtrace"))        ; 防止调试绕过

;; 网络全开（由 NE Extension 在更高层控制）
(allow network-outbound)
(allow network-inbound)

;; XPC / Mach 通信全开（开发工具需要）
(allow mach-lookup)

;; sysctl 白名单
(allow sysctl-read
  (sysctl-name "hw.ncpu")
  (sysctl-name "hw.memsize")
  (sysctl-name "kern.osversion"))
```

### 关键设计决策

#### 1. 为什么 deny `sandbox-exec` 自身？

如果不禁止，子进程可以用 `sandbox-exec -n no-sandbox` 启动一个不受限的子子进程，绕过所有限制。

#### 2. 为什么 `file-write-unlink` 需要特殊处理？

macOS 的 `rename()` 系统调用会对源路径做 `file-write-unlink` 检查。如果用户的工作目录是 `/Users/dev/project/src/`，要允许在此目录内移动文件，但不能允许删除工作目录的**祖先目录**：

```scheme
;; 防止通过 rename 将工作目录祖先移走
(deny file-write-unlink (subpath "/Users/dev/project"))
;; 允许工作目录内部的删除/移动
(allow file-write-unlink (subpath "/Users/dev/project/src"))
```

#### 3. 为什么网络不在这里拦截？

`sandbox-exec` 只能做全有/全无的网络控制（`allow/deny network-outbound`），无法按域名、端口、协议精细拦截。需要借助 NetworkExtension 的透明代理能力。

## 监控拦截事件

sandbox-exec 的 deny 事件会写入 macOS 统一日志：

```bash
# 通过 session tag 过滤当前沙盒的拦截日志
log stream --predicate 'eventMessage CONTAINS "SANDBOX_TAG_abc123"' --level error

# 或从持久化日志中查询
log show --predicate 'eventMessage CONTAINS "SANDBOX_TAG_abc123"' --last 1h
```

在代码中可以通过 `OSLogStore` API 程序化读取这些日志，实时收集拦截事件用于审计。

## 局限性与注意事项

| 限制 | 说明 |
|------|------|
| 官方 Deprecated | Apple 已将 `sandbox-exec` 标注为 deprecated，但目前无替代方案 |
| 不支持动态修改 | Profile 在进程启动时确定，运行中无法修改规则 |
| 路径注入风险 | 用户路径可能包含 `"` 或 `\`，必须转义 |
| last-match-wins | SBPL 规则是**最后匹配优先**，顺序很重要 |
| 无 UID/GID 隔离 | 沙盒进程与宿主共享同一用户身份 |

### 路径转义示例

```rust
/// 转义路径中的特殊字符，防止 SBPL 注入
fn escape_sbpl_string(s: &str) -> String {
    s.replace('\\', "\\\\").replace('"', "\\\"")
}
```
