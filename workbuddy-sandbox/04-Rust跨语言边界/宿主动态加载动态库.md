# 🟡 宿主动态加载动态库

> 📋 前置知识：[Rust cdylib 与 C ABI 最小示例](./Rust-cdylib与C-ABI最小示例.md)

## 1. 为什么要动态加载

动态加载（Dynamic Loading）指运行时再加载 `.dylib/.so/.dll`，而不是编译期强链接。

在 WorkBuddySandbox 中，macOS FileProvider Extension 动态加载 `libsandbox_ffi.dylib`，这样可以：

- 避免 Extension 编译期强依赖 Rust 产物；
- 在运行时检查动态库是否存在；
- 失败时降级到代理通道；
- 控制不同平台的加载路径。

## 2. 解决方案概览

| 平台 | 加载 API | 查找符号 |
|------|----------|----------|
| macOS/Linux | `dlopen` | `dlsym` |
| Windows | `LoadLibraryW` | `GetProcAddress` |
| .NET | `DllImport` / `NativeLibrary` | 方法声明或运行时查找 |
| Swift | `dlopen/dlsym` | `unsafeBitCast` 成函数指针 |

## 3. 方案对比

| 方案 | 优点 | 缺点 |
|------|------|------|
| 编译期链接 | 简单，类型更早检查 | 缺库时启动失败 |
| 运行时加载 | 可降级、可替换 | 符号签名错误到运行时才暴露 |
| IPC 替代 | 无 ABI 问题 | 性能和复杂度不同 |

## 4. 如何写一个最小版本

C 侧概念：

```c
void* lib = dlopen("./libmini.dylib", RTLD_NOW);
int (*add)(int, int) = dlsym(lib, "add");
printf("%d\n", add(1, 2));
dlclose(lib);
```

Windows 概念：

```cpp
HMODULE lib = LoadLibraryW(L"mini.dll");
auto add = (int (*)(int, int))GetProcAddress(lib, "add");
printf("%d\n", add(1, 2));
FreeLibrary(lib);
```

Swift 动态绑定时还要把 `UnsafeMutableRawPointer` 转成具体函数类型。

## 5. 常见坑

- **符号名不稳定**：Rust 必须 `#[no_mangle]`，否则找不到符号。
- **函数签名不一致**：宿主声明错一个参数类型就可能崩溃。
- **库路径问题**：App、Extension、命令行的运行目录不同。
- **签名问题**：macOS Hardened Runtime 下未签名动态库可能加载失败。
- **释放顺序**：不要在仍持有函数指针和对象时卸载动态库。
