# 🟢 WorkBuddySandbox 学习路线

> 📋 前置知识：能写基础 Rust/Swift/C# 中任意一种语言，能使用命令行运行程序。

## 1. 为什么需要一条学习路线

WorkBuddySandbox 不是单一技术点，而是一组底层能力组合：进程、权限、IPC、文件系统、动态库、网络、日志。学习顺序错了，就会出现两种问题：

- 只背术语，不会实现；
- 直接读工程源码，被平台细节淹没。

正确顺序应该是：**先学最小原语，再学平台机制，最后回到项目工程化实现**。

## 2. 解决方案概览

| 阶段 | 目标 | 产出 |
|------|------|------|
| 进程 | 控制一次命令执行 | `mini-runner` |
| 协议 | 让宿主和服务进程通信 | `mini-ipc` |
| 文件 | 让修改可丢弃、可落盘 | `mini-overlay` |
| 边界 | 让 Rust 被其他语言调用 | `mini-ffi` |
| 快照 | 让历史可回溯 | `mini-snapfile` |
| 网络 | 让连接可审计 | `mini-proxy` |
| 调试 | 让问题可定位 | 排障清单 |

## 3. 学习路线对比

| 路线 | 结果 |
|------|------|
| 直接读 WorkBuddySandbox | 容易知道“它做了什么”，但不会独立实现 |
| 只看 OS 文档 | 原理分散，难映射回项目 |
| 最小实现 -> 项目映射 | 既能写出来，也能读懂工程复杂度 |

本专区采用第三条路线。

## 4. 具体学习路径

### 第一步：写一个最小进程启动器

阅读：

- [从 fork/exec 到进程隔离](./01-进程与沙盒/从fork-exec到进程隔离.md)

你要能写：

```text
mini-runner run '{"executable":"/bin/bash","args":["-c","echo hi"],"cwd":"/tmp"}'
```

输出：

```json
{"exitCode":0,"stdout":"hi\n","stderr":""}
```

### 第二步：把 runner 做成服务进程

阅读：

- [NDJSON 协议从零实现](./03-IPC协议与服务进程/NDJSON协议从零实现.md)
- [Unix Domain Socket 服务端](./03-IPC协议与服务进程/Unix-Domain-Socket服务端.md)
- [Windows Named Pipe 服务端](./03-IPC协议与服务进程/Windows-Named-Pipe服务端.md)

你要能写：

```text
宿主 -> mini-sandboxd: {"command":"Ping","messageId":"1"}
宿主 <- mini-sandboxd: {"messageId":"1","success":true}
```

### 第三步：增加文件变更层

阅读：

- [覆盖层文件系统从零实现](./02-虚拟文件系统与投影/覆盖层文件系统从零实现.md)
- [Windows ProjFS 最小 Provider](./02-虚拟文件系统与投影/Windows-ProjFS最小Provider.md)
- [macOS FileProvider 最小 Provider](./02-虚拟文件系统与投影/macOS-FileProvider最小Provider.md)

你要能写：

```text
source/a.txt = old
overlay write a.txt = new
read a.txt -> new
discard a.txt
read a.txt -> old
```

### 第四步：理解 Rust 和宿主语言边界

阅读：

- [Rust cdylib 与 C ABI 最小示例](./04-Rust跨语言边界/Rust-cdylib与C-ABI最小示例.md)
- [宿主动态加载动态库](./04-Rust跨语言边界/宿主动态加载动态库.md)

你要能写：

```text
Rust 导出 add(a,b)
Swift/C#/C 运行时加载动态库并调用 add(1,2)
```

### 第五步：增加快照能力

阅读：

- [从文件拷贝到 Git 语义快照](./05-快照与变更管理/从文件拷贝到Git语义快照.md)
- [libgit2 最小快照系统](./05-快照与变更管理/libgit2最小快照系统.md)

你要能写：

```text
commit a.txt -> v1
modify a.txt
commit a.txt -> v2
checkout a.txt v1
```

### 第六步：增加网络审计

阅读：

- [代理与透明代理基础](./06-网络拦截/代理与透明代理基础.md)
- [macOS NetworkExtension 理解路径](./06-网络拦截/macOS-NetworkExtension理解路径.md)

你要能写：

```text
CONNECT example.com:443
规则 deny -> HTTP 403 + NetworkBlockRecord
```

### 第七步：建立排障证据链

阅读：

- [macOS 代码签名与 entitlements 实战](./07-调试与系统权限/macOS代码签名与entitlements实战.md)
- [底层日志与排障方法](./07-调试与系统权限/底层日志与排障方法.md)

你要能写：

```text
现象 -> 查进程 -> 查 IPC -> 查权限 -> 查规则 -> 查平台日志 -> 查结构化记录
```

## 5. 回到 WorkBuddySandbox 的总映射

| 你写的最小模型 | WorkBuddySandbox 对应模块 |
|----------------|---------------------------|
| `mini-runner` | `src/sandbox/src/executor/` |
| `mini-sandboxd` | `src/sandbox/src/bin/sandbox_cli.rs` + `src/sandbox/src/ipc/` |
| `mini-overlay` | `src/projfs/src/overlay.rs` |
| `mini-ffi` | `src/sandbox-ffi/` + `src/projfs/src/ffi.rs` |
| `mini-snapfile` | `src/snapfile/` |
| `mini-proxy` | `src/sandbox/src/network/proxy.rs` |
| 排障清单 | `src/logging/` + `scripts/build-mac.sh` + macOS entitlements |

## 6. 最终检验

如果你能不看 WorkBuddySandbox，独立写出这些最小模型，再回来读源码时就不会只是在“翻译项目实现”。你会知道每个复杂模块背后的底层原语是什么，以及工程代码是在补哪些边界情况。
