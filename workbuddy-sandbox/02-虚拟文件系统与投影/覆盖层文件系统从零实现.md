# 🟡 覆盖层文件系统从零实现

> 📋 前置知识：理解目录、文件读写、相对路径。

## 1. 为什么需要覆盖层

假设 Agent 要修改 `/repo/a.rs`。如果直接写源目录：

- 用户无法一键丢弃；
- 删除操作很难恢复；
- 多个 Agent 会话互相污染；
- 很难只同步部分文件。

覆盖层文件系统（Overlay Filesystem）的最小思想是：

```text
读文件：优先读 overlay，没有再读 source
写文件：只写 overlay，不碰 source
删文件：写入 deletion marker，不立刻删 source
落盘：把 overlay 变更同步回 source
丢弃：删除 overlay 和 deletion marker
```

## 2. 解决方案概览

| 方案 | 简述 | 适用场景 |
|------|------|----------|
| 全量复制 | 会话开始复制整个目录 | 小目录、实现简单 |
| copy-on-write overlay | 只复制被修改文件 | 大仓库、按需变更 |
| 系统 union mount | 内核/系统层合并 | Linux 容器等场景 |
| Git 快照 | 用对象库记录版本 | 需要版本语义 |

WorkBuddySandbox 的文件投影更接近 copy-on-write overlay，并配合 snapfile 做快照。

## 3. 方案对比

| 维度 | 全量复制 | CoW overlay | Git 快照 |
|------|----------|-------------|----------|
| 启动速度 | 慢 | 快 | 中 |
| 空间占用 | 高 | 低 | 低到中 |
| 删除表达 | 真实删除 | marker | Git tree |
| 学习成本 | 低 | 中 | 中 |

要先学 CoW overlay，因为它是理解 ProjFS/FileProvider 的基础。

## 4. 如何写一个最小版本

目录结构：

```text
session/
├── source/              # 原始目录，只读看待
└── overlay/
    ├── modified/        # 新增/修改文件
    └── deletions.json   # 逻辑删除列表
```

读文件伪代码：

```python
def read(path):
    if path in deletions:
        raise FileNotFoundError(path)
    if exists(overlay / "modified" / path):
        return read_file(overlay / "modified" / path)
    return read_file(source / path)
```

写文件伪代码：

```python
def write(path, data):
    ensure_parent(overlay / "modified" / path)
    write_file_atomic(overlay / "modified" / path, data)
    deletions.remove(path)  # 写入等于恢复删除
```

删除文件伪代码：

```python
def delete(path):
    remove_if_exists(overlay / "modified" / path)
    deletions.add(path)
    save_json_atomic(deletions)
```

## 5. 常见坑

- **路径逃逸**：`../../secret` 不能越过 source/overlay 根目录。
- **删除 marker 与新增冲突**：先删后写，要从删除列表移除。
- **原子写**：`deletions.json` 写一半崩溃会损坏状态，应使用临时文件 + rename。
- **目录删除**：删除目录要影响其子路径枚举。
- **大小写问题**：Windows 默认大小写不敏感，macOS 取决于磁盘格式。

## 7. 练习题：你要亲手实现什么

1. 写一个 `mini-overlay`，支持 `read/write/delete/list`。
2. 实现 `list_changes`：区分 `added/modified/deleted`。
3. 实现 `discard(path)`：只丢弃某个文件的变更。
4. 实现 `sync(path)`：把某个文件的 overlay 状态应用回 source。
