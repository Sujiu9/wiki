# 🟢 底层日志与排障方法

> 📋 前置知识：[macOS 代码签名与 entitlements 实战](./macOS代码签名与entitlements实战.md)

## 1. 为什么底层开发必须先找证据

底层问题经常表现为一句话：“没生效”。但“没生效”可能来自：

- 进程没启动；
- IPC 没连上；
- 权限没签进去；
- Extension 没被系统拉起；
- 规则没同步；
- 日志写到了另一个进程的目录；
- 操作被系统拒绝但业务没显示。

排障的第一原则：**不要猜，先找证据链**。

## 2. 解决方案概览

| 层级 | 你要确认什么 |
|------|--------------|
| 进程层 | `sandbox-cli` 是否启动，是否输出 `IPC_ADDRESS=` |
| 协议层 | NDJSON 请求和响应是否匹配 `messageId` |
| 权限层 | codesign/entitlements/App Group 是否正确 |
| 平台层 | FileProvider/NE/tsbx 是否真的启动 |
| 规则层 | allow/deny 是否同步到执行侧 |
| 记录层 | `FileBlockRecord` / `NetworkBlockRecord` 是否产生 |

## 3. 方案对比

| 方式 | 优点 | 缺点 |
|------|------|------|
| stdout/stderr | 启动早期可见 | GUI/Extension 场景不稳定 |
| 文件日志 | 持久化，便于收集 | 要知道路径 |
| 系统日志 | 能看到 sandbox/NE 事件 | 噪声大 |
| 结构化事件 | 可用于 UI 展示 | 需要链路已建立 |

WorkBuddySandbox 使用 Rust xlog、Swift XLog、系统 log stream、Extension 日志等多条路径。

## 4. 如何建立最小排障闭环

一个最小日志闭环包含：

```text
init_log(app_home)
write_info("starting sandbox-cli")
write_error("open session failed", err)
flush_on_exit()
```

排查 `runExecutable` 失败的顺序：

```text
1. sandbox-cli 是否启动？stdout 是否有 IPC_ADDRESS？
2. 宿主是否连上 IPC？Ping 是否成功？
3. OpenSession 是否成功？workingDir 是否正确？
4. runExecutable 是否返回 success=false？error 是什么？
5. 有无 FileBlockRecord / NetworkBlockRecord？
6. 平台日志里是否有 deny / NE / tsbx 记录？
```

## 5. 常见坑

- **只看一个日志文件**：Rust、主 App、Extension、系统日志可能分开。
- **日志目录来自 app_home**：不同启动参数会写到不同目录。
- **异步日志未 flush**：崩溃前最后几行可能没落盘。
- **系统日志太晚看**：`log stream` 是实时流，不是万能历史库。
- **把权限问题当业务问题**：先查签名和 entitlements。

## 7. 练习题：你要亲手实现什么

1. 写一个最小 xlog 替代品：启动时创建日志文件，退出前 flush。
2. 给 `mini-runner` 加日志：记录 cwd、env、exitCode。
3. 给 `mini-ipc-server` 加日志：记录 messageId、command、success。
4. 设计一个排障表：现象、第一证据、第二证据、可能原因。
5. 用一次故意失败的 sandbox-exec 访问，串起命令日志、deny 日志和结构化 block record。
